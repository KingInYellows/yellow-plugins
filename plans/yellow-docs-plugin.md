# Feature: yellow-docs Plugin — Documentation Generation and Maintenance

## Overview

A general-purpose Claude Code plugin that diagnoses and treats documentation
problems in any repository. The plugin detects project structure and adapts its
analysis and generation accordingly. Core capabilities: audit for gaps/staleness,
AI-assisted doc generation, context-aware Mermaid diagram generation, and
stale-doc refresh from code changes.

## Problem Statement

### Current Pain Points

1. **Docs go stale** — code changes ship but docs don't get updated, leading to
   drift between what the code does and what the docs say
2. **Missing docs** — features exist with no documentation at all
3. **No visual aids** — architecture and flows described in prose when Mermaid
   diagrams would be clearer
4. **Hard to discover** — docs exist but people can't find what they need

### User Impact

Both developers (building/maintaining code) and users (consuming
libraries/plugins) waste time reading outdated docs, reverse-engineering
undocumented features, and mentally modeling complex architectures that should
have diagrams.

### Business Value

Reduces onboarding friction, decreases support burden, and improves code
maintainability by making documentation a tractable, tool-assisted activity
rather than a manual chore.

## Proposed Solution

### High-Level Architecture

```

      generate.md       # /docs:generate — AI-assisted doc creation
      diagram.md        # /docs:diagram — Mermaid diagram generation
      refresh.md        # /docs:refresh — update stale docs from code changes
      setup.md          # /docs:setup — validate prerequisites
  agents/
```

### Key Design Decisions

1. **Output location contract**: Generated docs go alongside their related code
   by default. API docs next to source, architecture docs in `docs/`, diagrams
   inline in the doc they illustrate. All generated files include a provenance
   comment (`<!-- generated by yellow-docs at <commit> on <date> -->`).
   Override with `--output <path>` on any write command.

2. **Staleness detection algorithm**: Composite signal:
   - Source files in same directory or import graph modified since doc's last
     commit (git-blame based)
   - Doc age exceeding configurable threshold (default: 90 days)
   - Broken internal references (function/class names that no longer exist)

<!-- deepen-plan: external -->
> **Research:** A production-validated composite staleness formula from CodeScene
> and academic sources:
> `staleness_score = normalize(a*age_factor + b*effort_factor + c*cochange_factor + d*blame_shift_factor)`
> where age_factor = recency of source change vs doc update, effort_factor =
> normalized edit distance in referenced regions, cochange_factor = inverse of
> doc-code co-modification frequency, blame_shift_factor = fraction of referenced
> lines whose blame attribution changed. Practical thresholds: ignore temporal
> couples with <10 shared commits, ignore coupling strength <50%, ignore changesets
> with >50 files (refactor noise), flag as "likely stale" when score >0.7. Use
> `git blame -M -C` for move/copy detection across renames.
> See: [CodeScene Change Coupling](https://codescene.io/docs/guides/technical/change-coupling.html),
> [MDPI Co-modification Study](https://www.mdpi.com/2674-113X/5/1/11).
<!-- /deepen-plan -->

3. **Human review protocol**: All generated content goes through AskUserQuestion.
   User can approve, reject, or provide revision instructions. Up to 3 revision
   rounds before the agent asks if the user wants to write manually. For
   `/docs:refresh` with multiple stale files, review is per-file.

4. **Target/scope resolution**: Reserved keywords are lowercase: `readme`,
   `architecture`, `api-reference`. File paths must start with `./` or `/` or
   contain `/`. Omitted arguments default to repo root.

5. **Mermaid node limits**: Target 15–30 nodes for readability. Hard limit of 50.
   When scope exceeds 50 nodes, auto-summarize by collapsing leaf nodes into
   group nodes and warn the user. Override with `--max-nodes <n>`.

6. **Project detection priority** (most specific wins):
   `.claude-plugin/plugin.json` > `Cargo.toml` > `pyproject.toml`/`setup.py` >
   `go.mod` > `package.json` + `tsconfig.json` (TS) > `package.json` (JS).
   Multiple signals are all reported; the primary determines default doc
   strategy.

<!-- deepen-plan: codebase -->
> **Codebase:** In this repo, plugin manifests live at `.claude-plugin/plugin.json`,
> not at the project root. The detection logic should check `.claude-plugin/plugin.json`
> for Claude Code plugin detection. No shared utility libraries exist across plugins —
> project detection must be self-contained within yellow-docs.
<!-- /deepen-plan -->

<!-- deepen-plan: external -->
> **Research:** Use lockfile heuristics for package manager detection alongside
> manifests: `pnpm-lock.yaml` → pnpm, `yarn.lock` → yarn, `package-lock.json` → npm,
> `Cargo.lock` → cargo, `poetry.lock` → poetry, `go.sum` → go modules. For monorepo
> detection, check workspace configs: `pnpm-workspace.yaml`, root `package.json` with
> `workspaces`, `Cargo.toml` with `[workspace]`, `go.work`. Run detection per-package
> and aggregate. See: [antfu-collective/package-manager-detector](https://github.com/antfu-collective/package-manager-detector).
<!-- /deepen-plan -->

7. **Implementation order**: audit → generate → diagram → refresh (each builds
   on the previous; refresh depends on all three being stable).

### Trade-offs Considered

- **Generating alongside source vs. `docs/generated/`**: Alongside is more
  discoverable but risks accidental edits. Mitigated by provenance comments.
- **AST-based drift detection vs. git-blame only**: AST-based (symbol-level) is
  more precise but requires per-language parsers. Start with git-blame, add AST
  in a follow-up iteration.
- **Numeric health score vs. findings-only**: Include both. Score is secondary
  (`100 - (P1*15 + P2*5 + P3*1)`, floored at 0). Primary output is the
  findings list.

<!-- deepen-plan: external -->
> **Research:** Graph density matters more than raw node count. Density formula:
> `density = 2 * edges / (nodes * (nodes - 1))`. High-density (>0.3) becomes
> visually overloaded at ~50 nodes; low-density (≤0.3) remains readable up to ~100.
> Collapse strategies: (1) folder-depth collapse — summarize at directory depth N
> using subgraphs, (2) focus mode — show selected module + immediate neighbors
> only, (3) hierarchical drilldown — top-level overview + per-package details.
> Use stable node IDs (hashes) so diagram diffs are meaningful across regenerations.
> Sanitize special characters in labels (hex-encode parentheses, brackets).
> See: [mermaid-sonar](https://entropicdrift.com/blog/mermaid-sonar-complexity-analyzer/),
> [dependency-cruiser options](https://github.com/sverweij/dependency-cruiser/blob/main/doc/options-reference.md).
<!-- /deepen-plan -->

- **Smoke tests**: Run each command against this repo (yellow-plugins) and

- **Edge cases**: Test with empty repo, monorepo, non-git directory, repo with

- **Agent frontmatter**: Verify via `node scripts/validate-agent-authoring.js`

## Acceptance Criteria

1. `pnpm validate:schemas` passes with yellow-docs included
2. `/docs:audit` produces a structured findings report with coverage %, staleness
   findings, and gap findings for any git repo
3. `/docs:generate readme` produces a draft README from code analysis and
   presents it for human review before writing
4. `/docs:diagram architecture` generates a valid Mermaid flowchart from code
   structure analysis with ≤50 nodes
5. `/docs:refresh` identifies stale docs based on git history and generates
   per-file update diffs for human review
6. All commands gracefully degrade for unsupported project types
7. No generated content includes sensitive patterns (API keys, credentials)
8. Plugin works in any git repository, not just yellow-plugins

## Edge Cases and Error Handling

- **Empty repo**: Single summary finding, no enumeration
- **No git history**: Skip staleness, report gaps only
- **Monorepo**: Detect workspaces, aggregate per-package results
- **Binary/generated files**: Skip non-text files in scan
- **Existing docs at output path**: Show diff, ask to overwrite/merge
- **Diagram scope too large**: Auto-summarize, warn, offer `--max-nodes`
- **Unsupported language**: File-structure-only analysis with warning
- **Sensitive content in source**: Sanitize before including in generated docs
- **Git renames**: Use rename detection and history chaining to preserve
  doc-code mappings across file moves/renames
- **Symlinks and submodules**: Skip symlinks (security), optionally scan
  submodules with `--include-submodules` flag

## Security Considerations

- Never include file contents matching secret patterns in generated docs
- Respect `.gitignore` — never scan ignored paths
- Generated docs are drafts requiring human approval — no auto-write
- No external network calls — all analysis is local (git + file reads)

## Performance Considerations

- Cap audit findings at 50 per category to avoid noise in large repos
- Target execution times: audit <60s for 5k files, generate <30s per target,
  diagram <20s
- Use Glob/Grep for file discovery (not recursive Bash find)
- Start with git-blame staleness; defer AST-based symbol-level drift to v2

## Migration and Rollback

- **New plugin** — no migration needed, no existing state to preserve
- **Rollback**: Remove the plugin directory and marketplace entry; no other
  files are affected
- **No breaking changes** to any existing plugin
